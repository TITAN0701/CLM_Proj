# Task Writing Guide

此文件定義撰寫任務描述的規範，目的是：

1. **消除 AI 猜測** — AI 不需要推斷範圍、來源、目標，直接執行
2. **產出一致** — 相同類型的任務每次產出相同深度與格式
3. **深度足夠** — 任務描述說明「做到什麼程度」，不只說「做什麼」

適用範圍：`qa-workspace/current-task.md` 的任務清單，以及對話中直接給 AI 的指示。

---

## 核心結構：一條任務 = 動詞 + 對象 + 來源/目標 + 完成標準

```
{動詞} {對象}，來源：{從哪裡讀}，目標：{寫到哪裡}，完成時：{驗收條件}
```

**不是每條任務都需要四個欄位**，但缺少哪個就會出現對應的問題：

| 缺少什麼 | AI 的行為 | 後果 |
|---------|---------|------|
| 動詞不明確 | AI 自行判斷「做什麼」 | 可能只分析不執行，或執行錯誤動作 |
| 對象不具體 | AI 自行決定處理哪些項目 | 範圍過大或過小 |
| 來源不明 | AI 猜測資料從哪來 | 讀錯檔案，產出空資料 |
| 完成標準缺失 | AI 自行決定何時停止 | 深度不足就停，或做過頭 |

## 動詞規範

### 允許的動詞（語意明確）

| 動詞 | AI 的解讀 | 用法範例 |
|------|---------|---------|
| **補上** | 在現有資料中加入缺少的項目 | 補上 INT-149 的執行結果 |
| **更新** | 修改已存在的欄位值 | 更新 pipeline-state 的 pass 數 |
| **產出** | 從無到有建立新檔案或資料 | 產出 scenarios.md |
| **執行** | 跑腳本或指令並等待完成 | 執行 npm run sync:sheet |
| **同步** | 將本地資料推送到外部系統 | 同步 Google Sheets |
| **上傳** | 將檔案推送到雲端 | 上傳 Drive xlsx |
| **讀取** | 只讀，不修改 | 讀取 pipeline-state 並回報現況 |
| **驗證** | 檢查並回報是否符合規格 | 驗證 test-cases.json 格式是否正確 |
| **刪除** | 移除指定項目 | 刪除 BUG-AUTO-TC-UNKNOWN 開頭的 Bug |
| **修正** | 找到問題並修改到正確狀態 | 修正 scenarios.md 的 SC ID 格式 |
| **分析** | 讀取後給出判斷與說明，不修改檔案 | 分析哪些 TC 缺少 sc_ref |

### 禁止的動詞（語意模糊）

| 動詞 | 問題 | 改用 |
|------|------|------|
| **確保** | AI 不知道「確保」要做什麼動作 | 改用「驗證」或「修正」 |
| **處理** | 範圍不明，AI 自行決定做什麼 | 說清楚是「更新」還是「產出」 |
| **優化** | 沒有完成標準，AI 一直改 | 說清楚「優化哪個問題，改到什麼狀態」 |
| **幫我看** | 只是分析還是要修改？ | 改用「分析」（只看）或「修正」（要改） |
| **補齊** | 補哪些？補到什麼程度？ | 改用「補上 {具體項目}」 |
| **整理** | AI 會自行決定整理方式 | 說清楚「調整格式為 X」或「依 Y 排序」 |

## 來源與目標的寫法

### 來源（從哪裡讀資料）

任務涉及讀取資料時，必須明確指定來源，不可讓 AI 自行判斷。

```
✅ 正確：
- 補上 INT-139 的執行結果，來源：DevTools Console fetch 回傳的 500 錯誤截圖

❌ 錯誤（AI 會猜）：
- 補上 INT-139 的執行結果
```

常見來源寫法：

| 情境 | 寫法 |
|------|------|
| 從某個 JSON 檔讀 | `來源：qa-workspace/.pipeline-state.json` |
| 從截圖判斷 | `來源：上方截圖` / `來源：我提供的截圖` |
| 從 Excel 讀 | `來源：C:\...\RXCLM_QA_上線阻斷排程_單人版.xlsx` |
| 從對話內容讀 | `來源：上面的測試結果` |
| AI 自行執行取得 | `來源：執行 npm run check:alignment 的輸出` |

### 目標（寫到哪裡）

任務涉及修改或產出時，必須明確指定目標檔案或系統。

```
✅ 正確：
- 更新 INT-139 結果，目標：scripts/shared/blocking-list.json

❌ 錯誤（AI 會猜寫到哪）：
- 更新 INT-139 結果到報告
```

## 完成標準的寫法

完成標準告訴 AI「做到什麼程度才算完成」，防止 AI 做一半停下或做過頭。

### 格式

```
完成時：{可驗證的狀態描述}
```

### 範例

| 任務類型 | 完成標準寫法 |
|---------|------------|
| 補資料 | 完成時：blocking-list.json 的 INT-139 欄位 result 不為空白 |
| 執行腳本 | 完成時：npm run sync:sheet 回傳成功，無 error |
| 產出文件 | 完成時：scenarios.md 每個情境都有 Given/When/Then |
| 同步報告 | 完成時：Google Sheets 的「上線阻斷 Go-No-Go」分頁顯示最新資料 |
| 分析問題 | 完成時：列出所有缺少 sc_ref 的 TC ID 清單 |

## current-task.md 的完整格式

```markdown
## 任務清單

- {動詞} {具體對象}，來源：{從哪裡}，目標：{寫到哪裡}，完成時：{驗收條件}
- {動詞} {具體對象}

## 範圍限制（選填）

只處理以下 feature / 檔案：
- {具體路徑或 feature 名稱}

不處理：
- {明確排除的項目}

## 備註（選填）

- {任何會影響執行的上下文，例如：某個設定已知有問題、某個欄位格式特殊}
- 遇到錯誤時：{說明 AI 應該停下來回報，還是繼續跳過}
```

## 好/壞範例對照

### 範例 1 — 補測試結果

```
❌ 壞：
Test Reports 分頁後的資料都沒補到，請補上，且確保後續產出都能完整

問題：
- 「Test Reports 分頁後」不知道是哪些分頁
- 「補到」不知道資料來源
- 「確保後續產出完整」沒有完成標準，AI 會一直跑下去

✅ 好：
分析 Google Sheets 中 Failures、Integration Tests、Risk Notes、Bug Reports 四個分頁
為何沒有資料，來源：scripts/shared/sheet-definitions.js 的 loader 函式，
完成時：回報每個分頁的 loader 回傳筆數與問題原因
```

---

### 範例 2 — 更新 blocking-list

```
❌ 壞：
幫我把 INT-149 寫到報告上

問題：
- 「寫到報告上」不知道目標是哪個檔案
- 沒有說要寫哪些欄位
- 沒有說 INT-149 的結果是什麼

✅ 好：
更新 INT-149 執行結果，
來源：本次對話中 fetch verify-integrity 回傳 intact=true、verifiedRows=3434，
目標：scripts/shared/blocking-list.json 的 INT-149 欄位，
完成時：result="Pass"、actual 填入 API 回傳摘要、tester 填入 admin@local、date 填入今天日期
```

---

### 範例 3 — 同步報告

```
❌ 壞：
幫我上傳更新

問題：
- 不知道「上傳更新」是 sync:sheet、upload:drive、還是兩個都要
- 沒有說是否要先跑 check:alignment

✅ 好：
執行完整報告同步流程：
1. npm run check:alignment（若失敗停下來告知）
2. npm run sync:sheet
3. node scripts/sheets/upload-to-drive.js
完成時：三個指令都成功，回報 Drive 連結
```

---

### 範例 4 — 分析問題（只看不改）

```
❌ 壞：
幫我看報告所有分頁內容是否有對齊

問題：
- 「幫我看」不知道是只分析還是要修改
- 「對齊」定義不清楚

✅ 好：
分析 Google Sheets 每個分頁的 header 欄數是否與 loader 回傳的資料欄數一致，
來源：scripts/shared/sheet-definitions.js，
完成時：列出每個分頁的 header 欄數 vs 資料欄數，標記不一致的分頁，不修改任何檔案
```

## 任務拆分規則

一條任務只做一件事。以下情況必須拆成多條：

| 情況 | 拆法 |
|------|------|
| 「A 然後 B」 | 拆成兩條，分別寫完成標準 |
| 「A 如果成功再做 B」 | 拆成兩條，第二條加「前提：第一條完成且無 error」 |
| 「分析完再修正」 | 先寫「分析」任務（不改檔案），確認分析結果後再寫「修正」任務 |

### 範例

```
❌ 壞（一條包太多）：
- 檢查 scenarios.md 格式，修正有問題的，然後同步 Google Sheets

✅ 好（拆三條）：
- 分析所有 scenarios.md 的 SC ID 格式是否符合 S\d{2}-\d{2}，完成時：列出不符合的檔案與行號，不修改
- 修正上一步列出的 SC ID 格式問題，目標：各 feature 的 scenarios.md，完成時：所有 SC ID 符合格式
- 執行 npm run sync:sheet，完成時：成功無 error
```

## AI 執行前必須定義的事（Pre-flight）

AI 收到任務後，**執行第一個動作前**必須先確認以下五件事已明確。
若有任何一項不明確，**先列出不明確的項目問使用者，不得自行假設後執行**。

### 1. 動作是什麼（Action）

> 我要「做」什麼？是讀取、修改、產出、還是執行腳本？

- ✅ 明確：「更新 blocking-list.json 的 INT-139 欄位」
- ❌ 不明確：「把 INT-139 寫到報告」→ 必須問「是更新 blocking-list.json，還是同步到 Sheets？」

### 2. 對象是什麼（Target）

> 我要對「哪個檔案 / 哪個欄位 / 哪些 TC」動手？

- ✅ 明確：`scripts/shared/blocking-list.json` 的 `INT-139` 項目
- ❌ 不明確：「報告上的資料」→ 必須問「是 Google Sheets、Drive xlsx、還是本地 JSON？」

### 3. 資料從哪來（Source）

> 我要寫入的值，是從哪裡讀到的？

- ✅ 明確：來源是對話中的 fetch 結果截圖
- ❌ 不明確：「之前測試的結果」→ 必須問「是本次對話的哪個回傳值，還是某個檔案？」

若來源是截圖或對話內容 → AI 必須引用具體的值（數字、文字），不得只說「依截圖」就寫入

### 4. 完成的邊界在哪（Scope）

> 我做到哪裡停？有哪些項目不在這次範圍？

- ✅ 明確：只更新 INT-139，不跑 sync:sheet
- ❌ 不明確：「補上所有資料」→ 必須問「是哪些 INT 項目？還是全部空白的都補？」

### 5. 完成的判斷標準（Done Criteria）

> 我怎麼知道自己做完了？

- ✅ 明確：`result` 欄位不為空、腳本無 error
- ❌ 不明確：「確保資料完整」→ 必須問「完整的定義是什麼？所有欄位非空？還是特定欄位？」

---

### Pre-flight 自我宣告格式

AI 在執行前必須輸出以下宣告，讓使用者確認理解正確：

```
📋 執行前確認：
- 動作：{更新 / 產出 / 執行 / 分析}
- 對象：{具體檔案或欄位}
- 來源：{資料從哪裡來}
- 範圍：{做哪些，不做哪些}
- 完成標準：{怎麼判斷做完}
```

若以上任一項有「不明確」或「需假設」→ 改為：

```
⚠️ 執行前有以下項目需確認，請補充後再繼續：
- {不明確的項目 1}：{說明為什麼不明確，以及兩個可能的解讀}
- {不明確的項目 2}：...
```

**不得在有未確認項目的情況下繼續執行。**

## 備註區的使用規則

備註區放「影響 AI 判斷的上下文」，不是給人看的說明。

### 應該寫進備註的內容

- 某個設定已知有問題（例如：token 已過期，執行前先重新授權）
- 某個欄位格式特殊（例如：date 欄位用 YYYY-MM-DD，不是 timestamp）
- 遇到錯誤時的處理方式（停下來等指示 / 繼續跳過 / 自動重試）
- 本次任務與上次任務的關聯（例如：上次 INT-139 已確認是 bug，本次只寫入結果不需要再測）

### 不應該寫進備註的內容

- 「當發生任何錯誤時請告知」— 這是預設行為，不需要寫
- 任務本身的說明 — 應該寫在任務清單，不是備註
- 希望 AI 的語氣或風格 — 此專案不適用
